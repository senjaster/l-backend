# AGENTS.md — Правила и принципы кодовой базы L-Inspector Backend

Этот документ описывает ключевые архитектурные принципы и правила, действующие в данном репозитории. Он выведен исключительно из анализа реального кода.

---

## 1. Стек и окружение

- **FastAPI** — основной веб-фреймворк.
- **PostgreSQL** — единственная база данных, схема `lesiv`.
- **asyncpg** — основной драйвер (async). **psycopg2** — альтернативный драйвер для serverless-окружений (Yandex Cloud Functions / AWS Lambda через Mangum).
- **aiosql** — все SQL-запросы хранятся в `.sql`-файлах в `app/queries/`, загружаются через `aiosql.from_path()`. Никакого ORM нет.
- **Pydantic v2** — валидация данных и сериализация моделей.
- **pydantic-settings** — конфигурация через переменные окружения / `.env`.
- **Flyway** — миграции базы данных (`db/migrations/`).
- **Python venv** — зависимости управляются через `requirements.in` / `requirements.txt` (pip-compile).

---

## 2. Архитектура: слои

Код строго разделён на слои. Зависимости направлены только вниз:

```
Routers (app/routers/)
    ↓
Services / Dependencies (app/services/, app/dependencies/)
    ↓
Repositories (app/repositories/)
    ↓
SQL Queries (app/queries/*.sql)
    ↓
PostgreSQL (схема lesiv)
```

### 2.1 Routers

- Один файл на агрегат: `plant.py`, `equipment.py`, `inspection.py`, `defect.py`, `image.py` и т.д.
- Роутер создаёт экземпляр репозитория на уровне модуля (singleton): `plant_repo = PlantRepository()`.
- Роутер отвечает за: получение зависимостей (conn, permission_service, ownership_validator), вызов репозитория, обработку исключений (`ConcurrentModificationError` → 409, `ValueError` → 400).
- Мутирующие операции всегда оборачиваются в `async with conn.transaction()`.
- Бизнес-логика в роутерах минимальна — только оркестрация.

### 2.2 Repositories

- Один класс на агрегат: `PlantRepository`, `EquipmentRepository`, `InspectionRepository` и т.д.
- Репозиторий реализует паттерн **Aggregate Repository**: читает и сохраняет агрегат целиком вместе со всеми дочерними сущностями.
- Стандартный интерфейс: `get_by_id`, `get_all`, `save`, `delete`.
- Метод `save` реализует **оптимистичную блокировку** (см. раздел 5).
- Метод `save` принимает параметр `force: bool` — при `force=True` пропускает проверку `server_modified_at` и помечает лишние дочерние сущности как удалённые.
- Репозиторий **никогда не вызывается напрямую из другого репозитория** (кроме `OwnershipValidator`, который использует репозитории для чтения).

### 2.3 SQL Queries

- Все запросы — в файлах `app/queries/<aggregate>.sql`.
- Формат aiosql: `-- name: query_name(param1, param2)` + SQL.
- Суффиксы aiosql: `^` — возвращает одну строку, `!` — не возвращает строк (INSERT/UPDATE/DELETE), без суффикса — возвращает итератор строк.
- Запросы загружаются один раз на уровне модуля репозитория.
- При использовании psycopg2 запросы оборачиваются в `AsyncWrapper` для совместимости с `async/await`.

### 2.4 Models

- В большинстве агрегатов **одна и та же Pydantic-модель** используется и как ответ GET, и как тело PUT-запроса (например, `Plant`, `Equipment`, `Inspection`, `Defect`). Разделение на отдельную request-модель есть только у `Image` (`PutImageRequestBody` vs `Image`) — из-за полей, которые клиент не должен задавать напрямую (`upload_status`, `server_uploaded_at`).
- Агрегат содержит дочерние сущности как вложенные списки (например, `Plant` содержит `list[Facility]`, `Inspection` содержит `list[InspectionStep]`).
- Для списочных эндпоинтов `\all` используются облегчённые модели (`PlantListItem`, `InspectionListItem`) без дочерних сущностей.
- Ответ списочного эндпоинта '\all` всегда обёрнут в объект с ключом `items`, например: `PlantListResponse(items=[...])`.
- **Известное отклонение**: эндпоинты `/by_plant_id/{plant_id}` у агрегатов `equipment`, `inspection`, `defect`, `image` возвращают голый `list[XxxModel]` без обёртки. Это нарушение принципа, но исторически сложившееся поведение — не менять без необходимости.
- Поле `server_modified_at` присутствует во всех изменяемых агрегатах.
- Поле `is_deleted` присутствует почти во всех сущностях — используется **логическое удаление**, физического удаления нет (кроме `image`).

## 3. База данных

### 3.1 Схема

- Все таблицы находятся в схеме `lesiv`.
- Первичный ключ таблицы всегда называется `id` (кроме таблиц many-to-many у которых нет своего id)
- Внешний ключ должен называться `referenced_table_name_id`, например `plant_id`, `inspector_id`
- Ссылка на родителя в той же таблице должна называться `parent_id`
- Корни агрегата и их долчерние сущности используют UUID в качестве PK: `plant`, `facility`, `equipment`, `equipment_control_point`, `equipment_defect`, `inspection`, `inspection_step`, `image`. Агрегаты создаются в клиентских приложениях, поэтому использовать SERIAL невозможно.
- Справочники используют SERIAL в качестве PK: `inspector`, `sticker_type`, `equipment_type`, `facility_template`, `defect_type`. Справочники управляются централизовано.
- **Внешние ключи в БД создаются только внутри агрегата**, не между агрегатами. Ссылки между агрегатами — через поле без FK (например, `equipment.facility_id` ссылается на `facility`, но FK не объявлен).

### 3.2 Миграции

- Версионированные миграции: `V1__`, `V2__`, ... — применяются один раз.
- Повторяемые миграции: `R__` — применяются при каждом изменении (права доступа, функции, представления).
- Инструмент: Flyway (`db/flyway.conf`).

### 3.3 Права доступа

- Приложение работает от роли `l_app_role`.
- Права минимальны: справочники — только `SELECT`; агрегаты — `SELECT, INSERT, UPDATE`; `image` — полный CRUD; `log` — только `INSERT`.
- Физического `DELETE` у приложения нет (кроме `image`).

## 4. Аутентификация и авторизация

### 4.1 JWT + Refresh Token

- **Access token**: JWT RS256, короткоживущий (по умолчанию 15 минут). Payload содержит: `sub` (inspector_id), `dev` (device_id), `scope` (access_level).
- **Refresh token**: случайный opaque-токен (256 бит), хранится в БД в виде SHA-256 хэша.
- Реализована **ротация refresh-токенов**: при каждом обновлении старый токен отзывается, выдаётся новый.
- Реализовано **обнаружение кражи токена**: если отозванный токен используется повторно — вся цепочка токенов отзывается.
- Реализовано **окно повторного использования** (`reuse_lifetime_min`): защита от гонки при параллельных запросах.

### 4.2 Middleware

- `AuthMiddleware` применяется глобально ко всем маршрутам.
- Публичные пути (без аутентификации): `/`, `/docs`, `/redoc`, `/openapi.json`, `/auth/*`.
- Поддерживаются два заголовка: `Authorization: Bearer <token>` и `X-Auth-Token: <token>` (для Yandex Cloud, где `Authorization` перехватывается собственным механизмомо авторизации).
- После успешной аутентификации объект `Inspector` сохраняется в `request.state.current_user`.

### 4.3 Уровни доступа (AccessLevel)

Иерархия: `READ < INSPECT < MODIFY`.

| Уровень  | Разрешённые операции |
|----------|----------------------|
| `READ`   | Только GET-запросы |
| `INSPECT`| GET + создание инспекций, дефектов, изображений |
| `MODIFY` | Все операции, включая создание/изменение объектов и захват (claim) станции |

- Проверка уровня доступа: `permission_service.require_access_level(...)`.
- Уровень доступа хранится в БД в поле `inspector.access_level` и передаётся в JWT как `scope`.

### 4.4 Доступ к станциям (Plant Access)

- Каждый инспектор видит только те станции, к которым у него есть доступ (таблица `lesiv.inspector_plant_access`).
- При создании новой станции создатель автоматически получает к нему доступ.
- При захвате (claim) завода пользователь также получает к нему доступ.
- Анонимный пользователь (при отключённой аутентификации, `id=-1`) имеет доступ ко всем станциям.

## 5. Оптимистичная блокировка (Optimistic Locking)

- Реализована во всех изменяемых агрегатах: `Plant`, `Equipment`, `Inspection`, `Image`.
- Клиент при обновлении обязан передать `server_modified_at` — значение, полученное при последнем чтении.
- Сервер сравнивает `server_modified_at` клиента с текущим значением в БД (с точностью до миллисекунды через `truncate_to_milliseconds`).
- При несовпадении — возвращается **HTTP 409** с телом `ConflictError` (содержит `server_modified_at`, `client_modified_at`, список конфликтов `conflicts`, список лишних дочерних ID `extra_child_ids`).
- Дополнительная проверка: если на сервере есть дочерние сущности, которых нет в запросе клиента — тоже 409.
- Параметр `force=true` (query param) позволяет пропустить проверку и принудительно перезаписать данные (помечает лишние дочерние сущности как удалённые).
- Отключить оптимистичную блокировку можно через `disable_optimistic_locking=True` в конфигурации (только для тестов).

---

## 6. Пессимистичная блокировка (Claim / Pessimistic Lock)

- Перед изменением станции (`PUT /plant`) или оборудования (`PUT /equipment`) устройство обязано **захватить станцию** (`POST /plant/by_id/{id}/claim`).
- Захват привязан к `device_id` из JWT-токена.
- Захват разрешён если: завод не захвачен, захвачен тем же пользователем, или захват **устарел** (stale).
- **Устаревание захвата**: захват становится устаревшим каждый день в 03:00 по московскому времени (00:00 UTC). Реализовано в [`is_claim_stale()`](app/utils/claim_utils.py).
- При попытке изменить завод/оборудование без захвата или с захватом другим устройством — возвращается HTTP 409.
- Освобождение захвата: `POST /plant/by_id/{id}/release`.
- Claim/release обновляют `server_modified_at` для синхронизации с клиентами.

## 7. Синхронизация агрегатов

- API спроектирован для **мобильного клиента с офлайн-режимом**: клиент хранит локальную копию данных и синхронизирует её с сервером.
- Все списочные эндпоинты поддерживают параметр `modified_since` (datetime) — возвращают только записи, изменённые после указанного момента.
- Значение по умолчанию для `modified_since`: `1790-01-01` (константа `DEFAULT_MODIFIED_SINCE`) — возвращает все записи.
- Клиент отправляет агрегат целиком (включая все дочерние сущности) через `PUT`. Сервер синхронизирует дочерние сущности: добавляет новые, обновляет существующие, помечает удалёнными те, которых нет в запросе.
- **Запрещено "воровать" дочерние сущности**: если дочерняя сущность уже принадлежит другому агрегату — операция завершается ошибкой.

---

## 8. Логическое удаление

- Все сущности имеют поле `is_deleted: bool`.
- Физического удаления нет (кроме `image`).
- Удалённые записи возвращаются клиенту (клиент сам решает, как их отображать).
- Метод `delete` в репозитории выполняет `UPDATE ... SET is_deleted = true`.

---

## 9. Обработка ошибок

- Все ошибки БД (asyncpg и psycopg2) перехватываются глобальными обработчиками в [`app/exceptions.py`](app/exceptions.py).
- Нарушения ограничений БД (`ForeignKeyViolation`, `UniqueViolation`, `NotNullViolation`, `CheckViolation`) → HTTP 400 с телом `BaseError(type, message)`.
- Ошибки валидации Pydantic (`RequestValidationError`) → HTTP 422 с телом `BaseError`.
- Конкурентные изменения (`ConcurrentModificationError`) → HTTP 409 с телом `ConflictError`.
- Все ошибки логируются с контекстом (`path`, `method`, `error_type`).

## 11. Конфигурация

- Все настройки — через переменные окружения или файл `.env`.
- Класс [`Settings`](app/config.py) на базе `pydantic-settings`.
- Некоторые ключевые флаги:
  - `REQUIRE_AUTH` — включить/выключить аутентификацию (по умолчанию `True`).
  - `TRUST_INVALID_TOKENS` — принимать просроченные/отозванные токены (только для разработки/тестов).
  - `DISABLE_OPTIMISTIC_LOCKING` — отключить оптимистичную блокировку (только для тестов).
  - `DB_DRIVER` — `asyncpg` (по умолчанию) или `psycopg2` (для serverless).
- RSA-ключи для JWT: передаются напрямую через переменные окружения (`PRIVATE_KEY`, `PUBLIC_KEY`) или через пути к файлам (`PRIVATE_KEY_PATH`, `PUBLIC_KEY_PATH`).

## 12. Serverless-режим

- Приложение поддерживает два режима запуска:
  1. **Обычный**: `app/main.py` с lifespan (пул соединений инициализируется при старте).
  2. **Serverless**: `app/serverless.py` через Mangum (lifespan отключён, пул инициализируется при первом запросе).
- В serverless-режиме используется `psycopg2` с `ThreadedConnectionPool` (меньше соединений, синхронный драйвер).
- `AsyncWrapper` и `AsyncConnectionWrapper` обеспечивают совместимость синхронного psycopg2 с async-кодом FastAPI.

## 13. S3 / Изображения

- Изображения хранятся в S3 (Yandex Cloud Object Storage).
- Клиент получает **presigned URL** для прямой загрузки в S3 (минуя сервер).
- После загрузки Yandex Cloud отправляет callback на `POST /image/s3-upload-callback`, сервер обновляет `upload_status` в БД.
- Статусы загрузки: `UNKNOWN` (по умолчанию), `UPLOADED`, `MISSING`.
- Presigned URL для скачивания генерируется динамически при каждом GET-запросе изображения.

## 14. Правила именования и структуры

- Файлы: `snake_case`.
- Классы: `PascalCase`.
- SQL-таблицы и поля: `snake_case`.
- Маршруты: `/entity/all`, `/entity/by_id/{id}`, `/entity/by_plant_id/{plant_id}`.
- Каждый агрегат имеет свой файл в каждом слое: `models/`, `repositories/`, `routers/`, `queries/`.
- Новый агрегат регистрируется в `app/main.py` через `app.include_router(...)`.
- Словарь сущностей предметной области:
  - Plant = станция (имеется в виду power plant и электростанция)
  - Facility = энергообъект. Это здание на электростанции
  - Equipment с is_container=True это "папка"
  - Inspection = осмотр
  - Sticker = термоиндикаторная наклейка или просто ТИН

## 15. Тестирование

- Тесты находятся в `tests/`, конфигурация в `pytest.ini`.
- Запуск тестов: активировать `.venv` (`source .venv/bin/activate`), затем `pytest tests/`.
- Перед запуском тестов необходим файл `.env.test` с переменными `FLYWAY_URL`, `FLYWAY_USER`, `FLYWAY_PASSWORD`.

### 15.1 Тип тестов

- Все тесты — **интеграционные**: работают с реальной БД PostgreSQL через `TestClient` FastAPI.
- Unit-тестов нет. Моки не используются.
- Аутентификация в тестах отключена глобально: `settings.require_auth = False` в [`tests/conftest.py`](tests/conftest.py).

### 15.2 Структура тестового файла

- Один файл на агрегат: `test_plant.py`, `test_equipment.py`, `test_inspection.py` и т.д.
- В начале файла объявляется `PUT_BODY_TEMPLATE` — словарь с минимально валидным телом PUT-запроса для данного агрегата.
- Фикстуры UUID (`plant_id`, `facility_id`, `equipment_id` и т.д.) объявляются на уровне модуля и возвращают `uuid4()`.
- Фикстура `<aggregate>_data` собирает тело запроса из шаблона и UUID-фикстур через `deepcopy(PUT_BODY_TEMPLATE)`.
- Фикстура `client` определена в `conftest.py` и создаёт `TestClient(app)` с scope `function`.

### 15.3 Фикстуры и сидирование данных

- `seed_test_data` (scope=`session`, autouse) — запускает Flyway-миграции и загружает `scripts/init_test_data.sql` один раз за сессию.
- `seed_test_plant_and_facility` (scope=`function`) — создаёт тестовый завод и объект в БД напрямую через `asyncpg`. Используется как зависимость фикстуры `equipment_data` и аналогичных.
- `seed_test_equipment` (scope=`function`) — создаёт тестовое оборудование в БД напрямую через `asyncpg`.
- `grant_plant_access` (scope=`function`) — вспомогательная фикстура-фабрика для выдачи доступа к заводу конкретному инспектору.
- Сидирование данных в БД выполняется через прямое подключение `asyncpg.connect(settings.get_database_url())`, **не** через API.
- Вставки используют `ON CONFLICT ... DO NOTHING` для идемпотентности.

### 15.4 Правила написания тестов

- Каждый тест — независимый: создаёт собственные данные через API (`client.put(...)`) в начале теста.
- Тест не полагается на данные, созданные другим тестом.
- Имя теста начинается с `test_`, описывает сценарий: `test_create_plant`, `test_concurrent_modification_detected`.
- Каждый тест имеет docstring с кратким описанием сценария.
- Проверяется не только `status_code`, но и тело ответа (ключевые поля).
- При проверке списков используется `assert len(...) >= N` (другие тесты могут добавить данные), если только тест не изолирован по UUID.
- При проверке конкретного элемента в списке — поиск по ID через `next((x for x in items if x["id"] == ...), None)`.
- Для тестов оптимистичной блокировки: сначала создать агрегат, получить `server_modified_at` из ответа, затем передать его в следующий запрос.
- Для тестов с `force=true/false` — передавать как query-параметр: `client.put("/plant?force=true", json=...)`.
- Тесты на ошибки проверяют `response.json()["detail"]` и поле `type` в теле ошибки.

### 15.5 Тесты безопасности и прав доступа

- Тесты уровней доступа — в `test_access_level_security.py` и `test_permissions.py`.
- Для тестирования с конкретным токеном: создать токен через `AuthService().create_access_token(user_id, device_id, "MODIFY")` и передать в заголовке `Authorization: Bearer <token>`.
- Тесты с `TRUST_INVALID_TOKENS=True` — в `test_trust_invalid_tokens.py`.
- Тесты с отключённой оптимистичной блокировкой — в `test_optimistic_locking_disabled.py`.

### 15.6 Что покрывать тестами

Для каждого агрегата обязательно покрыть:

1. **CRUD**: создание (`PUT` нового UUID), чтение (`GET /by_id`), обновление (`PUT` с корректным `server_modified_at`), удаление (`DELETE` или логическое через `is_deleted=true`).
2. **Список**: `GET /all` возвращает `{"items": [...]}`, `GET /by_plant_id/{id}` возвращает голый список.
3. **Оптимистичная блокировка**: обновление с неверным `server_modified_at` → 409; с верным → 200.
4. **`force=true`**: лишние дочерние сущности помечаются удалёнными; `force=false` → 409 при наличии лишних.
5. **Логическое удаление**: `is_deleted=true` сохраняется и возвращается в GET.
6. **Запрет переноса дочерних сущностей**: попытка "украсть" дочернюю сущность → 400.
7. **`modified_since`**: фильтрация списка по дате изменения.
8. **404** для несуществующего ID.

## 16. Статическая типизация (Pyright) и aiosql-стабы

### 16.1 Общая схема

Проект использует **Pyright** (`typeCheckingMode: basic`) для статической проверки типов. Конфигурация — [`pyrightconfig.json`](pyrightconfig.json).

Поскольку aiosql генерирует методы динамически из `.sql`-файлов, Pyright не может вывести их сигнатуры самостоятельно. Для этого используется автогенерируемый стаб [`stubs/aiosql/queries.pyi`](stubs/aiosql/queries.pyi).

### 16.2 Генерация стаба

Стаб генерируется скриптом [`scripts/generate_aiosql_stubs.py`](scripts/generate_aiosql_stubs.py):

```bash
source .venv/bin/activate
python scripts/generate_aiosql_stubs.py
```

Скрипт читает все `.sql`-файлы из `app/queries/`, определяет тип операции каждого запроса через `aiosql.types.SQLOperationType` и генерирует соответствующие сигнатуры

### 16.3 Когда нужно перегенерировать стаб

Стаб **необходимо перегенерировать** после любого из следующих изменений:

- Добавлен новый `.sql`-файл в `app/queries/`
- Добавлен новый запрос (`-- name: ...`) в существующий `.sql`-файл
- Изменены параметры существующего запроса
- Изменён тип операции запроса (например, добавлен суффикс `^` или `!`)

Стаб **не нужно редактировать вручную** — он полностью перезаписывается при каждом запуске скрипта.

### 16.4 Известные ограничения
Сейчас каждый репозиторий загружает только один нужный ему файл с запросами и это позволяет иметь в разных файлах запросы с одинаковым именем без реальных конфликтов. Однако при создании стаба это будет выглядеть как конфликт, т.к. все запросы описыываются одним общим стабом. По возможности нужно использовать уникальные имена запросов.

Сейчас практически во всех файлах есть запрос get_by_id. Нужно следить, чтобы у него был только один параметр id - тогда все такие запросы будут выглядеть одинаково и pyright не будет выдавать ошибки.
